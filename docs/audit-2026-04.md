# Depth audit — 2026-04-30

Baseline measurement of `rakka` crates against upstream Akka.NET.
Used as the reference point for the 15-phase plan in
[`full-port-plan.md`](full-port-plan.md). Re-running this audit (via
`cargo xtask audit` from Phase 0) must not show regressions.

## Method

- Compared each rakka crate's `src/` LOC to the corresponding
  upstream Akka.NET module under `akka.net/src/`.
- Searched rakka source for placeholder sentinels: `panic!`,
  `unwrap`, `expect`, `unimplemented!`, `todo!`,
  `Box<dyn Any`, `__placeholder__`, `// stub`, `// placeholder`,
  `// TODO`, `// FIXME`.
- Read public types and reasoned about which protocol machinery
  exists vs which is "type signature only."

## Per-subsystem findings

### rakka-core

- 6 routers vs upstream's 20+ (`Random`, `RoundRobin`,
  `Smallest-Mailbox`, `Broadcast`, `ScatterGatherFirstCompletion`,
  `TailChopping`, `ConsistentHashing`, `BalancingPool` and their
  `Pool`/`Group` variants are missing).
- Dispatchers are a thin wrapper over `tokio::spawn`; no
  `pinned-dispatcher`, no `single-thread-dispatcher`, no throughput
  bound, no mailbox-batch knob.
- Mailboxes: only a default unbounded variant; no priority, no
  bounded, no deque-based.
- IO: 43-LOC TCP/UDP stub vs upstream's ~6.4k LOC `TcpManager`/
  `UdpManager` state machines.
- Stash: marker trait whose storage actually lives in `Context`.
- FSM: trait-only; no state-transition DSL macro.
- Coordinated Shutdown: phase enum present but most upstream
  phases are no-ops.
- Pattern: `ask`, `pipe_to`, `gracefulStop` exist; `CircuitBreaker`,
  `BackoffSupervisor` absent.

### rakka-cluster

- Data structures present: gossip envelope, membership, reachability,
  vector clock, member roles.
- All 5 SBR strategies coded (`KeepMajority`, `StaticQuorum`,
  `KeepOldest`, `KeepReferee`, `LeaseMajority`).
- **Missing entirely**: `ClusterDaemon`, active gossip
  dissemination loop, push/pull gossip protocol PDUs, leader
  election state machine, convergence detection,
  heartbeat sender/receiver actors, cluster events bus
  (`MemberUp`/`MemberLeft`/`UnreachableMember`/…) wired through
  `event_stream`. SBR strategies are unreachable from membership
  today.

### rakka-cluster-sharding

- 6 small files. `ShardCoordinator` is 42 LOC (in-memory only).
- **Missing**: `PersistentShardCoordinator` (akka.net's is ~5k LOC
  alone), `ShardAllocationStrategy` trait + implementations,
  rebalance algorithm, passivation, remember-entities, 3-phase
  handoff protocol, distributed allocation journal.

### rakka-streams

- Solid: linear `Source`/`Flow`/`Sink` ops (map / filter / take / skip
  / scan / grouped / concat / prepend / delay / throttle / map_async
  / map_async_unordered / intersperse / buffer / wire_tap / tick /
  unfold / repeat / cycle / from_future / from_receiver),
  fan-in/fan-out junctions (merge / merge_all / concat / zip /
  zip_with / zip_with_index / broadcast), framing
  (delimiter / length-field), IO (FileIO / Tcp), `KillSwitch`,
  `RestartSource`, `SourceQueue` with `OverflowStrategy`,
  `RunnableGraph`, `ActorMaterializer`.
- **Missing (~35 operators)**: `groupBy`, `splitWhen`, `splitAfter`,
  `prefixAndTail`, `flatMapPrefix`, `groupedWithin`, `batchWithin`,
  `idleTimeout`, `keepAlive`, `delay`, `delayWith`, `pulse`,
  explicit `.async` boundary, real `mapAsync(parallelism)` with
  ordering buffer, `Supervision.Decider`, `BroadcastHub`,
  `MergeHub`, `PartitionHub`, `partition`, `balance`, `unzip`,
  `interleave`, `extrapolate`, `expand`, `recover`, `recoverWith`,
  `recoverWithRetries`, `mapError`, `divertTo`, `watchTermination`,
  `monitor`, `log`, `addAttributes`, `named`, `StreamRefs`.
- No HTTP integration (akka-http is ~20k upstream LOC; carved out
  as separate `rakka-http` crate in the plan).

### rakka-persistence

- `PersistentActor` trait with command → events → state model.
- `Journal` + `SnapshotStore` plugin contracts.
- `AtLeastOnceDelivery` / `UnconfirmedDelivery`.
- In-memory journal + snapshot reference implementations.
- `PersistenceQuery` only covers `events_by_persistence_id`.
- TCK is 161 LOC vs upstream's 3,764 LOC.
- **Missing**: `Eventsourced` base trait, `ReceivePersistent`
  (closure-style API), `PersistentFSM`, `RecoveryPermitter`
  (bounded recovery concurrency), async snapshot save during
  normal operation, full `PersistenceQuery` operators
  (`events_by_tag`, `current_*`, `all_persistence_ids`,
  typed offsets).
- Storage backend crates (`-sql`, `-redis`, `-mongodb`, `-cassandra`,
  `-aws`, `-azure`) are placeholder skeletons.

### rakka-remote

- Solid: TCP transport, AkkaProtocol handshake, AkkaPdu framing
  (Associate/Disassociate/Heartbeat/Payload/Ack), `EndpointManager`
  state machine, `RemoteActorRefImpl`, `RemoteActorRefProvider`,
  `RemoteWatcher`, `RemoteSystemDaemon`, `AddressUidExtension`,
  `SerializerRegistry`, `RemoteMetricsExtension`,
  `FailureDetectorRegistry` with phi-accrual default,
  `ThrottleTransport` / `FailureInjectorTransport` /
  `TestTransport` adapters, `ClusterRemoteAdapter`,
  `RemoteRouterConfig`.
- **Missing**: full reader/writer task split per peer, quarantine
  lifecycle (`Quarantined → Tombstoned` with purge interval), TLS,
  message chunking for large payloads, send-queue with bounded
  backpressure, LRU caches for `ActorPath ↔ RemoteRef` and
  serializer-id ↔ manifest, fully-typed `RemoteProps` for
  `Deploy::remote`.
- **Sentinels**: `panic!("unexpected pdu …")` in
  `transport/tcp.rs`; `__placeholder__` address used during root
  initialization in `system.rs`; `WatcherStub` placeholder in
  `system_daemon.rs`.

### rakka-distributed-data

- 5 CRDTs (`GCounter`, `PNCounter`, `GSet`, `ORSet`, `LwwRegister`)
  in a `RwLock<HashMap>`-backed `Replicator`.
- **Missing**: `ORMap`, `LWWMap`, `PNCounterMap`, `Flag`,
  `ORMultiMap`; delta-CRDT propagation (deltas instead of full
  state); read/write consistency levels (`local`/`from(n)`/
  `majority`/`all`); durable storage backend; `Subscribe` API for
  per-key change notifications.
- Replicator must become a real actor (P-3) sharing state via the
  Phase 6 gossip transport.

### rakka-cluster-tools

- `ClusterSingletonManager` + `ClusterSingletonProxy` exist as
  thin `RwLock` wrappers around a holder reference.
- `DistributedPubSub` and `ClusterClient` are essentially empty
  stubs — no mediator, no topic gossip, no contact-point
  discovery.

### rakka-cluster-metrics

- 57-LOC snapshot holder (`RwLock<HashMap>`).
- **Missing**: CPU/heap/disk sampling, gossip dissemination,
  `AdaptiveLoadBalancingRouter`, threshold/health triggers.

### rakka-config

- 344 LOC. TOML-only loader with Akka-style key mirroring.
- **Missing**: HOCON parser, `include "file"` resolution,
  `${path.to.value}` reference substitution, `${?ENV_NAME}`
  env-var substitution.

### rakka-testkit

- ~192 LOC. `TestKit` (preconfigured system), `TestProbe`,
  basic `EventFilter`.
- **Missing**: rich matchers (`expect_msg_pf`,
  `expect_msg_class`, `expect_no_msg`, `expect_terminated`,
  `expect_all_of`, `receive_while`, `receive_n`,
  `fish_for_message`), `TestScheduler` for virtual time,
  `MultiNodeSpec` harness for cluster integration tests.

### rakka-serialization-hyperion

- 99 LOC Serde/bincode shim. By design — not a wire-compatible
  port of CLR Hyperion. Documented as such in `PORTING.md`.

## Idiomatic-Rust deltas

The audit also flagged places where rakka transliterates .NET
patterns instead of using Rust idioms. These are addressed by the
12 principles in [`idiomatic-rust.md`](idiomatic-rust.md) and the
Phase 1 + Phase 13 sweeps:

- `Box<dyn Any + Send>` for sender erasure in
  `MessageEnvelope::sender` and `Context::current_sender`.
- Pervasive `Any::downcast<T>()` in `Replicator`, `Extensions`,
  `ServiceContainer`, `Props`.
- `RwLock<HashMap>` hubs for `Replicator`, mediator,
  `ClusterSingleton`, coordination, discovery, DI, sharding
  coordinator — should be actors.
- Closure-boxed `Props` factory; `props!` macro will replace it.
- ~40 stray `eprintln!`/`println!` sites in library code.
- No compile-time supervision contract; `SupervisorOf<C>` will
  introduce one.

## Tracking

Phase 0 ships `cargo xtask audit` which records baseline counts
into `docs/reports/audit-2026-04.json`. CI fails any PR that
regresses on:

- Total `unwrap_used` / `expect_used` / `panic` / `todo` /
  `unimplemented` per crate.
- Total `Box<dyn Any` / `__placeholder__` / `// stub` /
  `// placeholder` per crate.
- Stray `println!` / `eprintln!` / `dbg!` outside test modules.

**History.** The original baseline (pre-Phase-1.B) showed 8
`Box<dyn Any>` sites. Phase 1.B removed 2 of them in `rakka-core`.
The Phase 4 testkit work added 12 mutex `lock().unwrap()` sites in
`test_scheduler.rs` / `multinode.rs`; these are the standard idiom
when poisoning is impossible (callers don't span panic windows) and
will be swapped for `parking_lot::Mutex` in Phase 13. The `rakka-
core` `Box<dyn Any>` site that remains is in `event/dead_letters.rs`
where the recipient genuinely doesn't know the type — a documented
"When to deviate" exception per [`idiomatic-rust.md`](idiomatic-rust.md).

The original baseline (pre-Phase-1.B, 2026-04-30T01:00) showed 8
`Box<dyn Any>` sites. After Phase 1.B removed the deprecated
`tell_with_sender` / `Context::sender_any` / `MessageEnvelope::
sender_any` shapes, 2 sites in `rakka-core` were eliminated. The
remaining `rakka-core` site is in `event/dead_letters.rs` — a dead
letter's message type is genuinely unknown to the recipient, so
that one `Box<dyn Any + Send>` is preserved with a justification
comment (the standard "When to deviate" pattern from
[`idiomatic-rust.md`](idiomatic-rust.md)).

Notes about the empirical scan vs the earlier qualitative read:

- `Box<dyn Any>` is **8 sites total**, not "pervasive." The pre-audit
  read overstated it; the actual concentration is in `rakka-remote`
  (4) and `rakka-core` (3). Phase 1 still removes them, but the
  blast radius is smaller than the qualitative report suggested.
- `rakka-persistence-tck`'s `unwrap()`/`expect()` count is high (17 +
  3) because the TCK is *test-shaped library code* — assertions
  flow through `unwrap`. Phase 11 replaces these with explicit
  `assert!`/`Result` flows so consumers don't inherit panics.
- `rakka-profiler`'s `println!`/`eprintln!` are intentional CLI
  output and are exempted via per-crate `print_stdout = "allow"`
  (see Phase 0 lint inheritance).
